#### 🎮 游戏模式

**模式一：算法竞技场**
两支队伍，每队最多 5 个单位，双方各自写代码在对称地图上对抗。胜利条件：摧毁对方基地，或占领地图中心点超过 60 秒。
- 单位只能感知 LiDAR 范围内的东西
- 超出通信范围的单位无法收到新指令
- 单位被击中后移动变慢（模拟硬件损坏）

**模式二：迷宫探索竞速**
单人模式，随机生成迷宫，写导航算法让单位最快找到出口。排行榜按完成时间排名，专门用来对比不同导航算法在不同地图类型上的表现。

**模式三：护送任务**
3 个战斗单位 + 1 个移动缓慢的「货物」单位，需要把货物从 A 护送到 B，途中有敌方巡逻单位（固定 AI）。难点在于同时处理护航、侦察和队形控制。

**模式四：SLAM 挑战**
单位在完全黑暗的地图里，不知道初始位置，需要自主探索建图并找到隐藏目标点。地图探索度实时显示，100% 探索完成挑战。

#### 🤖 单位可编程 API

玩家写代码时能调用的接口，设计上尽量贴近真实 ROS 2 API：

```typescript
// 传感器读取
unit.lidar.getRanges()           // 返回 360 个距离值数组，和真实 RPLidar 格式一致
unit.lidar.getNearest()          // 最近障碍物的距离和方向
unit.odometry.getPosition()      // 当前估算位置（含漂移误差）
unit.odometry.getTruePosition()  // 调试用，真实位置

// 运动控制（差速驱动约束，不能瞬间横移）
unit.setVelocity(v, omega)       // 线速度 + 角速度
unit.moveTo(x, y)                // 内置 A*，自动导航
unit.stop()

// 感知
unit.camera.getEntitiesInView()  // 视野内的敌我单位
unit.map.getOccupancyGrid()      // 当前已建立的占据栅格地图

// 多机通信
unit.broadcast(message)          // 广播给通信半径内的友军
unit.onMessage(callback)         // 收到消息的回调

// 事件
unit.on('obstacle_detected', callback)
unit.on('enemy_spotted', callback)
unit.on('reached_goal', callback)
```

#### 🔬 Three.js 可视化层（最有趣的部分）

**LiDAR 扫描可视化**
- 从单位位置向四周射出 360 条细线
- 线的颜色根据距离变化（近红远蓝）
- 检测到敌方单位时射线变黄并闪烁
- 视野区域是真实的不规则光锥，被墙遮挡就是黑暗

**粒子滤波位置云**
- 每个单位周围渲染 200 个半透明小点，表示位置不确定度
- 刚启动时粒子散布很广，随移动和感知更新逐渐收敛
- 视觉上直接体现「机器人不知道自己在哪」这个核心问题

**路径规划可视化**
- A* 搜索时实时渲染已扩展节点（灰色方格动画）
- 最终路径用发光绿线显示
- 势场法模式下渲染整个力场热力图

**占据栅格地图**
- 已探索无障碍：浅色
- 已探索有障碍：深色
- 未探索：黑色（战争迷雾）
- 不确定区域：半透明灰

**单位状态 HUD**
- 每个单位头顶小图标实时显示 FSM 当前状态
- 点击单位展开完整传感器数据面板
- 实时显示 `console.log` 输出，就像调试真实机器人

#### 🌍 进阶地图机制

随关卡解锁新的物理约束，逼迫玩家升级算法：

- **打滑地面**：轮子打滑，里程计漂移变大，需要更好的定位算法
- **动态障碍物**：移动的墙、来回踱步的中立单位，A* 不再够用
- **通信干扰区**：地图某些区域单位无法通信，必须提前规划自主行为
- **不同地形摩擦系数**：沙地 / 冰面 / 普通地面，影响运动控制参数

#### 📐 这个项目能学到的机器人知识点

**1. 传感器建模**

- **LiDAR 射线投射 (Ray Casting)**：真实 LiDAR 就是向周围发射 360 条激光记录距离，游戏里用 Three.js Raycaster 实现，数据格式和 RPLidar 输出完全一样：一个距离数组 `[d0, d1, ... d359]`
- **传感器噪声模拟**：给每条射线加高斯噪声，让感知不完美，直接对应概率机器人学基础
- **传感器盲区**：最小/最大检测距离、近距离反射率低导致数据丢失，让算法必须处理不完整信息

**2. 运动学模型**

- **差速驱动运动学**：左右轮速度 `(vL, vR)` → 线速度和角速度，公式 `v = (vR+vL)/2`，`ω = (vR-vL)/L`，单位不能瞬间横移
- **里程计漂移**：通过累计轮子转动估算位置，但误差会累积，走越远越不准，游戏里模拟这个漂移
- **运动约束**：最大线速度、最大角速度、加速度限制

**3. 路径规划**

- **Bug 算法族 (Bug0, Bug1, Tangent Bug)**：最简单的导航算法，碰墙绕行，能直观看到局限性
- **A* 路径规划**：地图是占据栅格，需考虑机器人体积（膨胀障碍物），实时可视化扩展过程
- **势场法 (Potential Field)**：目标产生引力，障碍物产生斥力，视觉漂亮但会陷入局部最小值，这个 bug 在游戏里会非常明显
- **DWA 动态窗口法**：考虑运动约束的实时避障，在速度空间采样可行轨迹选最优，这就是 ROS 2 Nav2 的默认局部规划器

**4. 定位与地图**

- **占据栅格地图**：把世界划分成格子，每格存储「有障碍物的概率」，直接对应 ROS 2 的 `nav_msgs/OccupancyGrid` 消息格式
- **粒子滤波定位**：用一堆粒子表示可能的位置分布，移动后扩散，传感器更新后收敛，这是 AMCL 算法的核心
- **SLAM 概念**：不知道地图也不知道自己在哪，同时解决两个问题（模式四的核心）

**5. 控制理论**

- **PID 控制器**：控制单位精确跟随路径，用 PID 控制转向误差；P 太大会振荡，太小会偏离，调参这件事在游戏里直接可感受
- **状态机 FSM**：探索 → 发现目标 → 追击 → 攻击 → 撤退，实时显示状态节点

**6. 多机器人系统**

- **队形控制**：领航者-跟随者模式，多单位保持相对位置移动
- **任务分配**：多目标多单位最优分配，匈牙利算法 / 拍卖算法
- **通信限制**：通信半径外无法协调，模拟真实 WiFi 信号衰减

## 🔀 语言过渡：Python → C++

> 阶段一结束后、阶段二开始前完成，约 1-2 周

### 为什么现在切换

阶段一用 Python 快速建立了对机器人算法的直觉。进入 ROS 2 后，所有核心节点、传感器驱动、Nav2 源码都是 C++，继续用 Python 会越来越受限——性能瓶颈、读不懂源码、写不了底层实时控制。嵌入式控制器（STM32 / ESP32）根本不跑 Python，C++ 是唯一选择。

**策略：不是抛弃 Python，而是明确分工**

| 用 Python 的场合 | 用 C++ 的场合 |
|----------------|--------------|
| 快速验证算法原型 | ROS 2 核心节点 |
| AI 推理脚本（PyTorch / ONNX） | 传感器驱动 |
| 数据分析、可视化脚本 | 实时控制回路 |
| 一次性工具脚本 | 嵌入式固件（STM32 / ESP32） |

### 过渡任务清单

- [ ] C++ 核心语法回顾（有游戏开发经验可快速过）
  - 类与对象、模板基础
  - STL 容器：`vector` / `map` / `queue`
  - 智能指针：`shared_ptr` / `unique_ptr`
  - CMake 构建系统基础
- [ ] 用 C++ 手动实现项目 1-B 的 FK 矩阵运算（不用任何库）

- [ ] 理解 ROS 2 C++ 节点结构：`rclcpp::Node`、回调函数、定时器、生命周期

### 推荐资源

### 🚀 项目 3-A：**可以网页遥控的"赛车"机器人** *(第19-24周)*

**项目描述：** 用树莓派 + 电机驱动板，造一台四轮小车。手机/浏览器打开一个你写的网页，就能实时控制它走动，看到它的摄像头画面。底层电机控制和 ROS 2 节点全用 C++ 写。

**硬件清单（约 ¥400-600）：**
- 树莓派 4B (4GB)
- L298N 电机驱动模块
- 4WD 小车底盘套件
- 18650 电池组
- 树莓派摄像头 v2

**C++ 在这个项目里写什么：**
- 电机驱动节点（C++ + `pigpio` 库）：订阅 `/cmd_vel` → 计算左右轮 PWM → 控制 L298N
- 里程计发布节点（C++）：读取编码器脉冲 → 计算位移 → 发布 `nav_msgs/Odometry`
- PID 速度控制器（C++）：让实际轮速跟随目标速度，消除打滑误差

**里程碑：**
- [ ] 树莓派系统配置 + SSH + WiFi + ROS 2 安装
- [ ] **用 C++ 写**电机驱动 ROS 2 节点，控制 L298N 实现前进/后退/转向
- [ ] **用 C++ 写**差速驱动运动学：`/cmd_vel` → 左右轮速度分配
- [ ] **用 C++ 写** PID 速度控制回路
- [ ] **用 C++ 写**里程计节点，发布 `Odometry` 话题
- [ ] 写 TypeScript + WebSocket 网页：虚拟摇杆 + 实时摄像头 MJPEG 流
- [ ] **挑战**：加 MPU6050 陀螺仪，C++ 节点读取 IMU 数据做姿态融合

**需要补的知识点：**
- GPIO / PWM 控制（C++ `pigpio` 库）
- H 桥电路原理（L298N）
- 编码器与里程计计算
- PID 控制器 C++ 实现
- `nav_msgs/Odometry` 消息格式
- 电源管理基础
- MJPEG 视频流

### 🚀 项目 3-B：**能「看到」并抓东西的机械臂** *(第25-32周)*

**项目描述：** 用低成本舵机搭一个 4-DOF 桌面机械臂（或买开源套件如 SO-ARM100），加一个摄像头，用 OpenCV + 颜色识别，让机械臂自动找到并抓取彩色方块。IK 算法用 C++ 实现，和阶段一 Python 版本形成对比。

**硬件清单（约 ¥800-1500）：**
- SO-ARM100 套件（开源机械臂）或自购 MG996R 舵机 × 4
- PCA9685 舵机驱动板
- 树莓派 / Jetson Nano
- USB 摄像头
- 手眼标定板（打印即可）

**C++ 在这个项目里写什么：**
- IK 求解器（C++）：手动实现几何法 IK，对比 Python 版的性能差异
- 舵机控制节点（C++ + I²C）：给 PCA9685 发送 PWM 命令
- MoveIt2 集成（C++）：用 ROS 2 的官方机械臂规划框架
- 轨迹平滑节点（C++）：在关节角度之间做插值，避免机械臂抖动

**里程碑：**
- [ ] **用 C++ 写** I²C 驱动，通过 PCA9685 控制多路舵机
- [ ] **用 C++ 重写**阶段一的 IK 算法，对比运行速度
- [ ] 集成 MoveIt2，用它做碰撞检测和轨迹规划
- [ ] **手眼标定**：将摄像头坐标系对齐到机械臂坐标系
- [ ] OpenCV 颜色检测：找到红色方块的 3D 位置（Python 可接受）
- [ ] 全流程：看到方块 → 计算抓取姿态 → C++ IK → 舵机执行 → 抓起来
- [ ] **挑战**：加入力反馈估算，防止把物体夹碎

**需要补的知识点：**
- I²C 协议 + C++ Linux I²C 编程
- 舵机 PWM 控制
- MoveIt2 C++ API
- 手眼标定 (Hand-Eye Calibration)
- 相机内参标定（张正友标定法）
- 轨迹插值算法（线性 / 三次样条）
